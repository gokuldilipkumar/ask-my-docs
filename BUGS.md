# Technical Debt / Known Shortcuts

Tracked per `.agent/workflows/plan.md` "Technical Debt Strategy" and surfaced during `/build`.
Format: `- [ ] <item>` — check off and remove once resolved, noting the fix in `PROJECT_HISTORY.md`.

## Ingestion (Block 1)

- [x] ~~Bold-only subsection-header heuristic misfires on inline emphasis~~ — **Fixed 2026-07-11** in the same session as the spot-check that found it. Real-corpus run showed the FAA handbook bolds inline field labels mid-sentence ("**Rule #1:** If you want to move...", "**Throttle:** increase power..."), and the old heuristic flagged every one of these as a subsection header, fragmenting the corpus into 1,850 chunks with a **29-token median** (target 400-600) — 60% of chunks were under 50 tokens. Fix: `TextSpan` now tracks `line_span_count` (spans sharing its physical PDF line); `detect_subsection_headers` requires a header to be the sole content of its line. Reproduced on a synthetic fixture (`test_does_not_flag_inline_bold_emphasis_sharing_a_line_as_header`) before fixing, per TDD. Re-run against the real handbook: 1,850 → 1,008 chunks, median token count 29 → 173.5, sub-50-token share 60% → 33%. Also incidentally fixed a chapter-title misattribution (a chapter title split across a styled multi-run line was being misparsed the same way).
- [ ] `apply_sliding_window` (Chunk 1.9) sizes windows by word count as a token-count proxy. Confirmed via a real handbook chunk: 600 words (exactly at the `max_tokens` word-count target) measured at **1,275 actual tokens** — more than double — because numeral-dense text ("9-37", "10-1") explodes into several BPE subword tokens each. Still open after the 2026-07-12 body-page-range fix: 143 of 617 chunks exceed 650 tokens. A proper fix needs the window to trim by actual `count_tokens()` measurement, not word count. (Originally cause 1 of the "chunk-size gap" item; cause 2, front/back-matter noise, is fixed — see below.)
- [x] ~~Front/back-matter (TOC, Glossary, Index) chunked like body content, polluting retrieval~~ — **Fixed 2026-07-12** after Block 2's spot-check produced real evidence (index page-number-list chunks ranking top-5, once #1, for 2 of 3 sample queries). Probing the real corpus showed it was worse than retrieval pollution: TOC pages repeat "Chapter N: Title" lines verbatim (duplicating every detected chapter header), and Index/Glossary alphabet headings ("V", "S") plus page-number labels ("12-12") were detected as subsection headers, fragmenting back matter into ~330 junk sections (chapter 18 held 402 of 951 chunks). Fix: config-driven `chunking.body_page_start`/`body_page_end` (0-based inclusive) filters spans **before** header detection; `config.yaml` sets 21–390 for this corpus — front matter 0–20 and Index 391–405 dropped, Glossary 372–390 **kept** (real definitional content; a glossary V-speed chunk now surfaces for the VMC/VSO sample query). Re-ingested: 951 → 617 chunks, median tokens 173.5 → 329, sub-50-token share 33% → 8%, chapter 18 402 → 84 chunks, zero index/TOC chunks in any spot-check top-5.
- [ ] `SIZE_RATIO_THRESHOLD = 1.15` (`src/ingest/headers.py`) is still an untuned guess — not revisited this session since the line-sharing fix was the higher-impact bug. Worth re-checking now that the dominant noise source (inline bold) is gone.
- [ ] `classify_page_label` (`src/ingest/page_labels.py`, Chunk 1.4) matches on text shape only, not footer position — not wired into `chunk_pdf`'s `printed_page_label` field yet. Needs a bbox-based "near bottom of page" check before it's trustworthy enough to surface in citations.
- [ ] Table extraction is unhandled — the design doc found zero `Table N-N.` caption matches anywhere in the corpus. No table-specific handling exists and this session's spot-check did not specifically inspect weight-and-balance/V-speed reference pages; still an open risk.
- [x] ~~V-speed subscript fragmentation~~ — **Fixed 2026-07-12** (found during Block 2's Chunk 2.3 spot-check). The handbook typesets V-speeds as "V" plus a smaller subscript span ("MC", "SO") that PyMuPDF extracts separately — and does *not* set its superscript flag on, so geometry is the only signal. Fix: `_join_subscript_spans` in `src/ingest/pdf_loader.py` merges a span into its predecessor when it is <0.8× the size, x-adjacent (≤1pt gap), and shifted below the base span's top edge; criteria verified against real span data before writing the synthetic RED fixture. Real-corpus result: bare-"V" spans 100 → 37 (all legitimate: "V = airspeed" formulas, glossary letter headings), "VMC" now intact in 62 spans, and the VMC/VSO sample query surfaces actual VMC content ("VMC is highly sensitive to bank angle...") instead of nothing.
- [ ] Low: the "group items by page_index into a dict" pattern (`dict.setdefault(...).append(...)`) is duplicated three times — `src/ingest/headers.py:27-29`, `src/ingest/chunker.py:37-39`, `src/ingest/chunker.py:94-98` — found during audit's fresh-eyes review. Each usage groups a different value shape (header titles, spans, figure refs), so a generic `group_by(items, key)` utility risks being a premature abstraction forced across three unrelated shapes; not extracted this session. Revisit if a fourth usage appears.
- [ ] Low: `apply_sliding_window` (`src/ingest/chunker.py`) takes a whole `RawSection` but reads only `section.text`, forcing every test to build a 6-field dataclass to window a string. Narrow the signature to `apply_sliding_window(text: str, ...)`; update call site in `chunk_pdf` and the three tests in `tests/ingest/test_chunker.py`. Relatedly, `chunk_pdf` takes 5 exploded chunking scalars while `hybrid_retrieve` takes a `RetrievalConfig` object — pick one convention (pass `ChunkingConfig`) and update `src/app/main.py`. Found during 2026-07-13 audit.
- [ ] Low: `chunk_pdf` (`src/ingest/chunker.py`) mixes orchestration with the figure-ref collection/attribution policy (refs attach only to `sequence == 0`) buried mid-function. Extract the figure-ref block (lines ~117-130) into a helper so the attribution rule is separately visible/testable. Found during 2026-07-13 audit.

- [x] ~~HIGH: `make_chunk_id` collides on repeated section titles (617 rows / 590 unique ids)~~ — **Fixed 2026-07-13.** `page_index_start` joined the hash inputs (stable across re-ingestion for a fixed PDF, unlike an occurrence counter, which shifts when header detection changes earlier in a chapter), and `chunk_pdf` now *counts* the uniqueness invariant, raising `ValueError` on any collision instead of shipping ambiguous ids. The check fired immediately on the real corpus and exposed a second bug (below). After both fixes: re-ingested 617 → 612 chunks (5 bogus glossary sections absorbed), **612/612 unique ids in both indexes, id sets identical**. Spot-check re-run: retrieval stable, and a previously-colliding "Common errors ... crosswind" chunk is now distinctly retrievable (reranked #3 for the crosswind query).
- [x] ~~Low: `group_into_sections` matches header spans by text equality~~ — **Fixed 2026-07-13**, promoted from latent to real by the collision check: glossary V-speed notation leaves stray standalone letter spans ("V" + "FE" on separate PDF lines, beyond the within-line subscript join) whose text equals the page's alphabet heading, spawning 5 bogus same-page "V" sections on p388 (plus "E"/"M" cases). Header spans now match on `(title, font_size, is_bold)` per page — stray spans are body-sized, real headings are not. Synthetic RED fixture reproduced the exact real-corpus colliding id before the fix.
- [ ] Low: `chunk_pdf` (`src/ingest/chunker.py`) inlines the corpus-wide chunk-id uniqueness check (Counter + collision raise) at the end of an already-multi-job function (grouping, windowing, figure-ref attachment, chunk assembly). Extract to `_assert_unique_chunk_ids(chunks)` — a post-condition check is a distinct concern from building a chunk and becomes independently testable without running the whole PDF pipeline. Found during 2026-07-13 chunk-id-fix audit.
- [ ] Low: the "chunk_id must be unique" invariant is enforced in two places with no shared reference tying them together — `chunk_pdf` (`src/ingest/chunker.py`) raises on collision at write time; `get_chunk_texts` (`src/ingest/vector_index.py`) explicitly declines to assume uniqueness at read time and keeps an "arbitrary row wins" fallback for duplicates. If the write-side guarantee is ever loosened or tightened, nothing forces the read-side comment/logic to be revisited in step. Consider a shared docstring/constant cross-reference, or a test pinning the read-path duplicate-handling behavior. Found during 2026-07-13 chunk-id-fix audit.

## Config (Block 0)

- [ ] Low: `Settings` yaml/env paths are cwd-relative (`yaml_file="config.yaml"`, `src/config/settings.py`). Running the CLI from any directory other than the repo root **silently ignores** `config.yaml` — meaning the corpus body-page-range filter wouldn't apply and front/back matter would be ingested. Anchor the path to the project root or fail loudly when the yaml is absent. Decide before Block 4's serving layer. Found during 2026-07-13 audit.
- [ ] Low: every default in `src/config/settings.py` is restated verbatim in `config.yaml` (`min_tokens: 400`, `rrf_k: 60`, ...) — two sources of truth that can drift. Pick one: code defaults with yaml carrying only corpus-specific overrides (as `body_page_start/end` already do), or yaml as the single source with required fields. Found during 2026-07-13 audit.
- [ ] Low: `get_settings()` (`src/config/settings.py`) is an uncached passthrough whose name implies the standard `@lru_cache` idiom — a reader will assume memoization that isn't there. Note: naive `@lru_cache` would break CLI tests that vary env vars between in-process invocations; either cache deliberately (with invalidation story) at Block 4's serving layer or delete it and call `Settings()` directly in `src/app/main.py`. Found during 2026-07-13 audit.

## Retrieval (Block 2)

- [ ] Low: `hybrid_retrieve` (`src/retrieval/hybrid.py`) reloads the BM25 index from disk on every call. Fine for tests and spot-check scripts; wrong lifetime for a serving path (per-request disk read + pickle parse). Split index loading from query execution (load once, retrieve many) when Block 4's API layer lands. Found during 2026-07-13 audit.

- [ ] Low: the embedding model name is a hardcoded constant (`_MODEL_NAME`, `src/ingest/vector_index.py:8`) while the reranker model is config-driven (`rerank.model`) — an asymmetry flagged by the 2026-07-13 Block 3 audit as violating "config-driven, no magic numbers". Caveat the reviewer missed: the embed model is baked into the index at build time, so config-swapping it without re-ingesting silently breaks search (unlike the reranker, which is swap-safe at query time). If moved to config, the ingest CLI must record the model used and `search_vector` must refuse to query an index built with a different model. Decide at Block 4.
- [ ] Low: query-time functions live in the ingest package — `search_vector` and `get_chunk_texts` (`src/ingest/vector_index.py`) and `search_bm25`/`load_bm25_index` (`src/ingest/bm25_index.py`) are imported by `src/retrieval/hybrid.py`, inverting the package mental model (readers look for search in `retrieval/`, find it in `ingest/`). If moving: relocate the search/fetch functions to `src/retrieval/`, keep `build_*` in `ingest/`, update imports in `src/retrieval/hybrid.py`, `src/app/main.py`, `tests/ingest/test_bm25_index.py`, `tests/ingest/test_vector_index.py`, `tests/retrieval/test_hybrid.py`. Decide when Block 4 composes the pipeline (both index modules are touched then anyway). Found during 2026-07-13 Block 3 audit.

## Rerank (Block 3)

- [ ] **Rerank latency on real chunks is ~5.3s per query** (20 candidates, median 329 tokens, CPU) — vs the design doc's ~130ms/16-candidate claim, which holds only for short passages (probe: 63ms/16 short pairs; passage length dominates). At ~5s the reranker is the whole serving-path latency budget. Decide at Block 4: reduce `retrieval.top_n` (20 → 10 halves it), set an explicit CrossEncoder `max_length`, batch tuning, or accept it for a CLI-first project. Related: chunks over the model's 512-token input limit are **silently truncated during scoring** — same oversized-chunk population as the open token-accurate-windowing item (143 chunks > 650 tokens).

## Eval (Block 6)

- [ ] **HIGH, real-corpus finding, not fixed**: `answer_question`'s generation (Block 4) fabricated content on the first real eval run. For q4 ("What are the common errors during a crosswind takeoff?"), the model's answer opened with "According to the handbook, the most common errors..." and listed 8 specific numbered items, while citing only a single chunk (`0c98a0e1acc1577a`) that contains no such enumerated list — confirmed absent from the corpus entirely by Chunk 6.3's ground-truth audit (every comparable maneuver has a "Common errors in the performance of X" bulleted chunk; crosswind takeoff does not). The answer-quality judge caught this directly: "the answer fabricates the existence of a formal errors list and presents inferences as direct handbook content." This is a citation-scoped-generation failure Block 5's citation verification does *not* catch (the single cited chunk's content is itself relevant/supportive by Block 5's per-citation definition — the fabrication is in the *prose*, not an unsupported citation). Needs its own TDD fix to `prompts/answer_v1.md` (e.g. an explicit instruction not to present synthesized/inferred content as if directly quoted or enumerated from the source) plus a regression test reproducing this exact case. Deferred, not fixed ad hoc mid-Block-6, since Block 6's scope was building the harness that catches this, not immediately patching every generation-quality bug it finds.
- [ ] Real-corpus finding, not fixed: q1 (VMC vs. VSO) scored `correct=True` on the very first real end-to-end run and `correct=False` on an immediate re-run of the identical question (`GenerationConfig` has no `temperature` field, so generation uses the Anthropic SDK's default rather than a fixed low value) — the two runs' answers genuinely differed in which VSO details they included. Concrete evidence that answer correctness itself is non-deterministic across runs, not just retrieval/rerank (as Block 4/5's Q3 finding suggested) — motivates deciding whether `generation.temperature` should be added and set low/zero for reproducibility, at the cost of losing whatever quality benefit default sampling provides. Decide before this harness's numbers are trusted as a tight regression gate.
- [ ] Real-corpus finding, not fixed: q7 (FAA Wings Program) scored `complete=False` despite perfect retrieval (`recall_at_k=mrr=ndcg=1.0`) — the answer was factually accurate but didn't match the specific framing/analogy language written into its `reference_notes` (the "professional athletes/medical professionals" analogy, the "alternative pathway to the flight review" framing). This may be a `reference_notes` calibration issue as much as a system gap: writing a reference rubric that names a *specific* analogy from the source text turns "did you get the facts right" into "did you also match the source's specific rhetorical framing," a stricter and different bar than intended. Revisit `reference_notes` wording (favor required *facts*, not required *phrasing*) if this pattern repeats across more eval runs.
- [ ] `eval.cache_path`'s sqlite cache is keyed on `(question_id, config_hash)`, and `config_hash` only hashes `Settings` fields — it has no way to detect that the *code* computing a cached value changed (found when fixing the `mrr` bug above: the cache would have silently kept serving the stale pre-fix `mrr=0.0` for q8 indefinitely, since no `Settings` field changed). Worked around manually this session by deleting the cache file before the corrected re-run. A real fix needs a code-version component in the hash (e.g. a package/module version string bumped on eval-logic changes) — not built this session, low urgency until this harness runs unattended in CI (Block 8).
