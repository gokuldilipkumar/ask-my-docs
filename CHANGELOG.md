# Changelog

All notable changes to this project are documented here. Format loosely follows [Keep a Changelog](https://keepachangelog.com/).

## [Unreleased]

### Added
- `uv`-managed Python 3.11 project scaffolding (`src/`-layout packages: `ingest`, `retrieval`, `rerank`, `generate`, `citations`, `eval`, `observability`, `config`, `app`).
- Config-driven `Settings` (pydantic-settings): `config.yaml` + `.env` layering, required-secret validation, no magic numbers ‚Äî chunk size, RRF weights, rerank/citation/eval thresholds all config-driven from the start.
- PDF ingestion pipeline (PyMuPDF): text-span extraction with font metadata, chapter-header detection, subsection-header detection (font-size/bold, sole-line-placement), figure-caption extraction, deterministic content-hash chunk IDs, subsection-boundary-first chunking with a token-aware sliding-window fallback (400-600 tokens, 15% overlap) for oversized sections.
- BM25 sparse index (`bm25s`) and vector index (LanceDB + `bge-small-en-v1.5`) build/search, both with build‚Üíreload‚Üísearch round-trip tests.
- `ingest` CLI command (typer) wiring the full pipeline end to end.
- Ingested the real FAA Airplane Flying Handbook (406 pages): 617 chunks across both indexes (after front/back-matter filtering; was 1,008 at first ingest).
- Hybrid retrieval (Block 2): `reciprocal_rank_fusion` (config-weighted RRF, pure function) and `hybrid_retrieve` orchestrating the existing BM25 + vector search; real-corpus spot-check confirms flipping fusion weights via config visibly reorders results (design-doc acceptance criterion).
- Config-driven body page range (`chunking.body_page_start`/`body_page_end`): spans outside the body range are dropped before header detection. For this corpus (21-390): front matter/TOC and the back-of-book Index are excluded; the Glossary is kept as retrievable definitional content.
- Cross-encoder reranking (Block 3): `rerank()` scores `(query, chunk_text)` pairs with a config-named cross-encoder (`rerank.model`, default ms-marco-MiniLM-L-6-v2) and truncates to `rerank.top_k`; `enabled=False` is a passthrough that preserves fused order and provably never loads the model. Candidates travel as `(chunk_id, text)` pairs ‚Äî no parallel lists to mismatch. Real-corpus spot-check: toggling rerank changed the top-5 on all three sample queries, demoting a keyword-soup V-speed-list chunk off the VMC/VSO top-5 in favor of actual VMC content.
- `get_chunk_texts(db_path, chunk_ids)`: id‚Üítext lookup on the existing LanceDB table (the reranker's join between `hybrid_retrieve`'s IDs and scoreable text); raises `KeyError` on missing IDs rather than silently returning partial results.
- Grounded generation (Block 4): `generate_answer` calls `claude-sonnet-5` via `client.messages.parse(output_format=GeneratedAnswer)` for structured `{answer_text, citations: [chunk_id]}` output ‚Äî no hand-parsed JSON. SDK-native retry/timeout (`client.with_options(max_retries=, timeout=)`); extended thinking explicitly disabled (short extraction-and-cite task doesn't need it). Empty/no retrieved chunks short-circuits to an honest "I don't have that" answer without ever calling the API. `answer_question` orchestrates retrieve ‚Üí rerank ‚Üí generate, reusing one `get_chunk_texts` fetch for both the reranker's candidates and the final prompt. Versioned prompt template (`prompts/answer_v1.md`, `PROMPT_VERSION` constant for future trace/eval attribution). New `live_api` pytest marker (skipped unless `RUN_LIVE_API_TESTS=1`) for the first tests that call the real, paid Anthropic API. Real-corpus spot-check: 7/8 design-doc sample questions cited correctly; the out-of-scope question (helicopter autorotation) returned an honest non-answer with empty citations.
- Citation verification (Block 5): `verify_citations` (`src/citations/verify.py`) batches every cited chunk into one `claude-haiku-4-5-20251001` structured-output call (`VerificationResult.verdicts: list[CitationVerdict]`) instead of one call per citation ‚Äî one round-trip regardless of citation count. Zero citations short-circuits to `coverage=1.0`, `low_confidence=False` without touching the API. A verdict missing for a cited chunk_id defaults to unsupported (fail-safe stripping, not silently kept). Malformed/truncated judge output raises a clear `RuntimeError` naming `citations.max_tokens` instead of a raw `pydantic.ValidationError`. `answer_with_verified_citations` (`src/citations/pipeline.py`) composes Block 4's `answer_question` with `verify_citations`, fetching cited-chunk text only when there are citations to verify. Versioned judge prompt (`prompts/verify_v1.md`, `PROMPT_VERSION = "verify_v1"`). Real-corpus spot-check: all 8 design-doc sample questions landed at `coverage=1.0`; the judge adds ~1s (roughly 3-5%) to total query latency on top of generation.
- Evaluation harness (Block 6): golden dataset of the 8 design-doc sample questions (`eval/golden/questions.yaml`) with LLM-labeled + user-reviewed ground-truth `relevant_chunk_ids` and hand-authored `reference_notes`. Pure retrieval metrics (`src/eval/retrieval_metrics.py`: Recall@k, MRR, nDCG, deterministic, zero API cost). A ground-truth relevance-labeling judge (`src/eval/relevance.py`, one-time/offline) distinct from the repeatable answer-quality judge (`src/eval/judge.py`) which scores `correct`/`complete` independently per question ‚Äî completeness is a new metric targeting the previously-logged "answered half the question" gap. sqlite response cache (`src/eval/cache.py`) keyed on `(question_id, config_hash)` to avoid re-spending during eval-harness debugging. Git-commit + prompt-version-tagged baseline storage (`src/eval/baseline.py`) with tolerance-band comparison, not hard cutoffs. `run_eval` (`src/eval/pipeline.py`) composes all of the above, gated to only `reviewed=True` golden questions. First real baseline saved (2026-07-14): `mean_recall_at_k=0.616, mean_mrr=0.906, mean_ndcg=0.783, mean_coverage=1.000, correctness_rate=0.750, completeness_rate=0.500`.

### Fixed
- Subsection-header detection was misclassifying inline bold emphasis (e.g. "**Rule #1:** ...") as standalone headers, fragmenting 60% of real-corpus chunks to under 50 tokens against a 400-600 target. Headers now require sole-line placement, not bold/size alone. Re-ingestion: 1,850 ‚Üí 1,008 chunks, median chunk size 29 ‚Üí 173.5 tokens.
- `apply_sliding_window` accepted a `min_tokens` floor it never enforced, allowing arbitrarily tiny trailing-window fragments on long sections. Trailing under-floor windows now merge into the previous window.
- V-speed subscripts ("V" + smaller "MC"/"SO" spans, extracted separately by PyMuPDF with no superscript flag set) fragmented terms like "VMC" and broke BM25 matching for V-speed queries. Subscript spans now join their base span geometrically; "VMC" survives extraction intact in 62 spans (was 0).
- TOC pages repeat "Chapter N: Title" lines verbatim (duplicating every detected chapter header) and Index/Glossary headings fragmented back matter into ~330 junk sections that polluted retrieval top-5s. Fixed by the body-page-range filter above. Corpus quality after both fixes: 951 ‚Üí 617 chunks, median tokens 173.5 ‚Üí 329, sub-50-token share 33% ‚Üí 8%.

- `reciprocal_rank_fusion` silently dropped rankings beyond the weights list length (`zip` without `strict=True`) ‚Äî a mis-sized config would quietly turn hybrid retrieval into single-index retrieval. Now raises `ValueError` on length mismatch. Found by the audit's fresh-eyes review; Block 2's four happy-path tests all passed over it.
- `get_chunk_texts`'s scan limit silently truncated results when the table holds duplicate chunk_ids (its fail-loud KeyError caught this on first real-corpus contact); chunk IDs are now SQL-escaped in the LanceDB filter string so no ID can corrupt the query.
- **Duplicate chunk IDs (HIGH)**: `make_chunk_id` collided on section titles the handbook repeats per maneuver (617 chunks, 590 unique IDs; "Common errors..." √ó5 on one ID). `page_index_start` now joins the hash, and `chunk_pdf` counts the uniqueness invariant, raising on collisions instead of shipping ambiguous IDs.
- That collision check immediately exposed a second bug on the real corpus: text-equality header matching let stray glossary V-speed letter spans spawn bogus sections under same-page alphabet headings ("V" √ó5 on p388). Headers now match by font signature (title, size, bold). Re-ingested: 617 ‚Üí 612 chunks, **612/612 unique IDs in both indexes, identical ID sets**; previously-colliding chunks are now distinctly retrievable.
- The generation prompt's structured `citations` field could include a chunk id the model examined and rejected, not just ones supporting the answer (found via a live-API test: the model cited a chunk while explicitly explaining it didn't answer the question). Prompt now states the `citations` field's own contract explicitly ‚Äî mentioning what an excerpt covers is not citing it.
- `generate_answer` raised a raw `pydantic.ValidationError` three SDK frames deep when a detailed answer exceeded `max_tokens`, truncating the model's structured JSON mid-string. Found by the Block 4 real-corpus spot-check on a genuinely long-answer question. Now catches the error and raises a clear `RuntimeError` naming `max_tokens`; default raised 1024 ‚Üí 2048.
- `config.example.yaml`'s `generation:` section had silently drifted from `config.yaml` (stale model name, a dropped retry field, two missing new ones) ‚Äî found by the Block 4 audit's fresh-eyes review, since no test or `Settings` load path ever reads the example file. Synced.
- `mrr` (`src/eval/retrieval_metrics.py`) was missing the empty-`relevant_ids` vacuous-truth guard `recall_at_k`/`ndcg` both already had, scoring `0.0` instead of `1.0` for a deliberately-zero-relevant-docs question (q8). Found on the first real end-to-end eval run.
- `eval.pipeline._evaluate_one` retrieved and reranked twice per golden question ‚Äî once directly for retrieval metrics, once again inside `answer_with_verified_citations` for generation ‚Äî doubling the ~5.3s/query real-corpus rerank cost for no correctness gain. Found during the Block 6 audit. Fixed by reusing one retrieve+rerank pass for both metrics and generation (`generate_answer`/`verify_citations` called directly instead of the composed `answer_with_verified_citations`).
- `judge_answer` and `label_relevance` duplicated the entire `client.with_options(...).messages.parse(...)` + truncation-error-translation logic. Extracted to `call_structured_judge` (`src/eval/llm_call.py`).

### Known issues (tracked in `BUGS.md`)
- Sliding-window sizing still uses word count as a token-count proxy, which undercounts badly on numeral-dense text; 143 of 617 chunks exceed 650 tokens.
- `SIZE_RATIO_THRESHOLD` (subsection-header font-size heuristic) is still an untuned constant.
- Printed page labels (roman numeral / chapter-relative) are classified but not yet wired into chunk metadata.
- Table extraction is unhandled and unverified against the real corpus.
- `Settings` loads `config.yaml` relative to the cwd ‚Äî running the CLI outside the repo root silently skips the corpus body-page-range filter (plus six smaller entropy items from the 2026-07-13 audit).
- Rerank latency on real chunks is ~5.3s per 20-candidate query on CPU (design claim of ~130ms holds only for short passages); chunks past the model's 512-token input limit are silently truncated during scoring. Block 4 decision item.
- Real-corpus spot-check found one design-doc sample question (short-field vs. soft-field takeoff) only half-answered ‚Äî a retrieval/rerank surfacing gap, not a generation bug. Input for Block 6's golden dataset. (2026-07-14 spot-check: the same question answered both halves correctly on this run ‚Äî non-deterministic retrieval/rerank, not confirmed fixed; no regression test pins either outcome.)
- `generate_answer` discards the Anthropic response's token usage; Block 7's cost calculator needs it.
- Citation verification (Block 5) only checks the citations `GeneratedAnswer` already listed, not full per-claim faithfulness of `answer_text` ‚Äî an uncited hallucinated sentence isn't caught. `citations.judge_model`/`judge_temperature` duplicate `eval.judge_model`/`judge_temperature`'s default without sharing config. `citations/pipeline.py` re-fetches cited-chunk text rather than reusing Block 4's internal fetch.
- **HIGH**: Block 6's first real eval run caught generation fabricating an 8-item "According to the handbook" errors list (question 4) that isn't present in its single cited chunk or anywhere else in the corpus ‚Äî a citation-scoped-generation failure Block 5's per-citation faithfulness check doesn't catch, since the cited chunk itself is genuinely relevant. Not yet fixed.
- `GenerationConfig` has no `temperature` field (defaults to the Anthropic SDK's default) ‚Äî the same question (q1, VMC vs. VSO) scored `correct=True` on one run and `correct=False` on an immediate re-run, real answer-correctness non-determinism, not just retrieval/rerank variance.
- The eval response cache's `config_hash` only hashes `Settings` fields, not code logic ‚Äî a metrics-code bug fix does not invalidate stale cached results with the same config.
