# Changelog

All notable changes to this project are documented here. Format loosely follows [Keep a Changelog](https://keepachangelog.com/).

## [Unreleased]

### Added
- `uv`-managed Python 3.11 project scaffolding (`src/`-layout packages: `ingest`, `retrieval`, `rerank`, `generate`, `citations`, `eval`, `observability`, `config`, `app`).
- Config-driven `Settings` (pydantic-settings): `config.yaml` + `.env` layering, required-secret validation, no magic numbers — chunk size, RRF weights, rerank/citation/eval thresholds all config-driven from the start.
- PDF ingestion pipeline (PyMuPDF): text-span extraction with font metadata, chapter-header detection, subsection-header detection (font-size/bold, sole-line-placement), figure-caption extraction, deterministic content-hash chunk IDs, subsection-boundary-first chunking with a token-aware sliding-window fallback (400-600 tokens, 15% overlap) for oversized sections.
- BM25 sparse index (`bm25s`) and vector index (LanceDB + `bge-small-en-v1.5`) build/search, both with build→reload→search round-trip tests.
- `ingest` CLI command (typer) wiring the full pipeline end to end.
- Ingested the real FAA Airplane Flying Handbook (406 pages): 617 chunks across both indexes (after front/back-matter filtering; was 1,008 at first ingest).
- Hybrid retrieval (Block 2): `reciprocal_rank_fusion` (config-weighted RRF, pure function) and `hybrid_retrieve` orchestrating the existing BM25 + vector search; real-corpus spot-check confirms flipping fusion weights via config visibly reorders results (design-doc acceptance criterion).
- Config-driven body page range (`chunking.body_page_start`/`body_page_end`): spans outside the body range are dropped before header detection. For this corpus (21-390): front matter/TOC and the back-of-book Index are excluded; the Glossary is kept as retrievable definitional content.
- Cross-encoder reranking (Block 3): `rerank()` scores `(query, chunk_text)` pairs with a config-named cross-encoder (`rerank.model`, default ms-marco-MiniLM-L-6-v2) and truncates to `rerank.top_k`; `enabled=False` is a passthrough that preserves fused order and provably never loads the model. Candidates travel as `(chunk_id, text)` pairs — no parallel lists to mismatch. Real-corpus spot-check: toggling rerank changed the top-5 on all three sample queries, demoting a keyword-soup V-speed-list chunk off the VMC/VSO top-5 in favor of actual VMC content.
- `get_chunk_texts(db_path, chunk_ids)`: id→text lookup on the existing LanceDB table (the reranker's join between `hybrid_retrieve`'s IDs and scoreable text); raises `KeyError` on missing IDs rather than silently returning partial results.
- Grounded generation (Block 4): `generate_answer` calls `claude-sonnet-5` via `client.messages.parse(output_format=GeneratedAnswer)` for structured `{answer_text, citations: [chunk_id]}` output — no hand-parsed JSON. SDK-native retry/timeout (`client.with_options(max_retries=, timeout=)`); extended thinking explicitly disabled (short extraction-and-cite task doesn't need it). Empty/no retrieved chunks short-circuits to an honest "I don't have that" answer without ever calling the API. `answer_question` orchestrates retrieve → rerank → generate, reusing one `get_chunk_texts` fetch for both the reranker's candidates and the final prompt. Versioned prompt template (`prompts/answer_v1.md`, `PROMPT_VERSION` constant for future trace/eval attribution). New `live_api` pytest marker (skipped unless `RUN_LIVE_API_TESTS=1`) for the first tests that call the real, paid Anthropic API. Real-corpus spot-check: 7/8 design-doc sample questions cited correctly; the out-of-scope question (helicopter autorotation) returned an honest non-answer with empty citations.
- Citation verification (Block 5): `verify_citations` (`src/citations/verify.py`) batches every cited chunk into one `claude-haiku-4-5-20251001` structured-output call (`VerificationResult.verdicts: list[CitationVerdict]`) instead of one call per citation — one round-trip regardless of citation count. Zero citations short-circuits to `coverage=1.0`, `low_confidence=False` without touching the API. A verdict missing for a cited chunk_id defaults to unsupported (fail-safe stripping, not silently kept). Malformed/truncated judge output raises a clear `RuntimeError` naming `citations.max_tokens` instead of a raw `pydantic.ValidationError`. `answer_with_verified_citations` (`src/citations/pipeline.py`) composes Block 4's `answer_question` with `verify_citations`, fetching cited-chunk text only when there are citations to verify. Versioned judge prompt (`prompts/verify_v1.md`, `PROMPT_VERSION = "verify_v1"`). Real-corpus spot-check: all 8 design-doc sample questions landed at `coverage=1.0`; the judge adds ~1s (roughly 3-5%) to total query latency on top of generation.
- Evaluation harness (Block 6): golden dataset of the 8 design-doc sample questions (`eval/golden/questions.yaml`) with LLM-labeled + user-reviewed ground-truth `relevant_chunk_ids` and hand-authored `reference_notes`. Pure retrieval metrics (`src/eval/retrieval_metrics.py`: Recall@k, MRR, nDCG, deterministic, zero API cost). A ground-truth relevance-labeling judge (`src/eval/relevance.py`, one-time/offline) distinct from the repeatable answer-quality judge (`src/eval/judge.py`) which scores `correct`/`complete` independently per question — completeness is a new metric targeting the previously-logged "answered half the question" gap. sqlite response cache (`src/eval/cache.py`) keyed on `(question_id, config_hash)` to avoid re-spending during eval-harness debugging. Git-commit + prompt-version-tagged baseline storage (`src/eval/baseline.py`) with tolerance-band comparison, not hard cutoffs. `run_eval` (`src/eval/pipeline.py`) composes all of the above, gated to only `reviewed=True` golden questions. First real baseline saved (2026-07-14): `mean_recall_at_k=0.616, mean_mrr=0.906, mean_ndcg=0.783, mean_coverage=1.000, correctness_rate=0.750, completeness_rate=0.500`.
- Observability (Block 7): `Tracer`/`SpanHandle` protocols (`src/observability/tracer.py`) with a `NoOpTracer` safe default and a `LangfuseTracer` wrapping the real `langfuse==4.14.0` SDK's OTel-based `start_as_current_observation` (`get_tracer(settings)` returns `NoOpTracer()` whenever `observability.langfuse_enabled=False` or Langfuse credentials are unset — config-driven passthrough, same shape as `rerank.enabled=False`). `hybrid_retrieve` now opens 3 nested spans (bm25/vector/fusion), `rerank` opens 1, `generate_answer`/`verify_citations` each open 1 `as_type="generation"` span carrying model/usage/cost — all via a new optional `observability` parameter, zero behavior change when omitted. One shared `ObservabilityContext`, constructed once in `answer_with_verified_citations` and threaded through the whole call chain, so a single live query produces one nested trace instead of several disconnected ones. Pure `calculate_cost` (`src/observability/cost.py`) prices `(model, input_tokens, output_tokens)` against a config-driven `price_table` (placeholder figures, not yet verified against real Anthropic pricing); a sqlite-backed daily running cost total (`src/observability/daily_cost.py`, mirrors `eval/cache.py`'s pattern) with a warn-only budget check (`observability.daily_cost_cap_usd`, never blocks calls). `report_usage` (`src/observability/usage.py`) is the one function every real API call site calls to record cost — including both eval judges via `call_structured_judge`, cost-tracking only, no dedicated spans (eval judges run in the offline eval flow, not the traced query-time path).
- CLI completion + CI (Block 8): `query` command answers a question end-to-end via `answer_with_verified_citations`, printing the answer, citations, coverage/low-confidence flag, and the daily running cost total. `run_eval`/`_evaluate_one` gain a `retrieval_only` mode computing only `recall_at_k`/`mrr`/`ndcg` (zero Anthropic API calls, sqlite cache bypassed) — `EvalResult`/`EvalRunResult`'s answer-quality fields become `Optional`, `None` (not a sentinel) when skipped; `compare_to_baseline` skips any metric field `None` on either side. `eval` command runs the harness (full or `--retrieval-only`), prints per-metric PASS/FAIL vs. the latest baseline, exits non-zero on any regression beyond `eval.tolerance`, and supports `--save-baseline`. `.github/workflows/cheap-gate.yml` runs the test suite + `eval --retrieval-only` on every push (zero real API cost); `.github/workflows/nightly-eval.yml` runs the full judge-based `eval` daily + on manual trigger, saving a new baseline on success. Repository pushed to a real GitHub remote (`github.com/gokuldilipkumar/ask-my-docs`); `data/index/` (the built BM25 + LanceDB corpus index) is now committed so CI never needs the 273MB source PDF. Both workflows verified passing on real GitHub Actions infrastructure.
- Portfolio deliverables (Block 9, final planned block): `README.md` with a Mermaid architecture diagram, an explained rationale for hybrid retrieval/RRF/reranking/citation verification, a quickstart section where every command's output is real and freshly captured (not hand-typed), an observability section with a real Langfuse trace screenshot, and CI status badges. `docs/eval-report.md`: full methodology, current baseline metrics table, a real reproducible regression (`retrieval.top_n: 20 → 1`, real gate failure, exit code 1 — found after two other real attempts didn't reproduce a regression), and a "what this harness has already caught" section citing real bugs from project history plus a second, live-captured instance of generation non-determinism found while writing the report itself.

### Fixed
- Subsection-header detection was misclassifying inline bold emphasis (e.g. "**Rule #1:** ...") as standalone headers, fragmenting 60% of real-corpus chunks to under 50 tokens against a 400-600 target. Headers now require sole-line placement, not bold/size alone. Re-ingestion: 1,850 → 1,008 chunks, median chunk size 29 → 173.5 tokens.
- `apply_sliding_window` accepted a `min_tokens` floor it never enforced, allowing arbitrarily tiny trailing-window fragments on long sections. Trailing under-floor windows now merge into the previous window.
- V-speed subscripts ("V" + smaller "MC"/"SO" spans, extracted separately by PyMuPDF with no superscript flag set) fragmented terms like "VMC" and broke BM25 matching for V-speed queries. Subscript spans now join their base span geometrically; "VMC" survives extraction intact in 62 spans (was 0).
- TOC pages repeat "Chapter N: Title" lines verbatim (duplicating every detected chapter header) and Index/Glossary headings fragmented back matter into ~330 junk sections that polluted retrieval top-5s. Fixed by the body-page-range filter above. Corpus quality after both fixes: 951 → 617 chunks, median tokens 173.5 → 329, sub-50-token share 33% → 8%.

- `reciprocal_rank_fusion` silently dropped rankings beyond the weights list length (`zip` without `strict=True`) — a mis-sized config would quietly turn hybrid retrieval into single-index retrieval. Now raises `ValueError` on length mismatch. Found by the audit's fresh-eyes review; Block 2's four happy-path tests all passed over it.
- `get_chunk_texts`'s scan limit silently truncated results when the table holds duplicate chunk_ids (its fail-loud KeyError caught this on first real-corpus contact); chunk IDs are now SQL-escaped in the LanceDB filter string so no ID can corrupt the query.
- **Duplicate chunk IDs (HIGH)**: `make_chunk_id` collided on section titles the handbook repeats per maneuver (617 chunks, 590 unique IDs; "Common errors..." ×5 on one ID). `page_index_start` now joins the hash, and `chunk_pdf` counts the uniqueness invariant, raising on collisions instead of shipping ambiguous IDs.
- That collision check immediately exposed a second bug on the real corpus: text-equality header matching let stray glossary V-speed letter spans spawn bogus sections under same-page alphabet headings ("V" ×5 on p388). Headers now match by font signature (title, size, bold). Re-ingested: 617 → 612 chunks, **612/612 unique IDs in both indexes, identical ID sets**; previously-colliding chunks are now distinctly retrievable.
- The generation prompt's structured `citations` field could include a chunk id the model examined and rejected, not just ones supporting the answer (found via a live-API test: the model cited a chunk while explicitly explaining it didn't answer the question). Prompt now states the `citations` field's own contract explicitly — mentioning what an excerpt covers is not citing it.
- `generate_answer` raised a raw `pydantic.ValidationError` three SDK frames deep when a detailed answer exceeded `max_tokens`, truncating the model's structured JSON mid-string. Found by the Block 4 real-corpus spot-check on a genuinely long-answer question. Now catches the error and raises a clear `RuntimeError` naming `max_tokens`; default raised 1024 → 2048.
- `config.example.yaml`'s `generation:` section had silently drifted from `config.yaml` (stale model name, a dropped retry field, two missing new ones) — found by the Block 4 audit's fresh-eyes review, since no test or `Settings` load path ever reads the example file. Synced.
- `mrr` (`src/eval/retrieval_metrics.py`) was missing the empty-`relevant_ids` vacuous-truth guard `recall_at_k`/`ndcg` both already had, scoring `0.0` instead of `1.0` for a deliberately-zero-relevant-docs question (q8). Found on the first real end-to-end eval run.
- `eval.pipeline._evaluate_one` retrieved and reranked twice per golden question — once directly for retrieval metrics, once again inside `answer_with_verified_citations` for generation — doubling the ~5.3s/query real-corpus rerank cost for no correctness gain. Found during the Block 6 audit. Fixed by reusing one retrieve+rerank pass for both metrics and generation (`generate_answer`/`verify_citations` called directly instead of the composed `answer_with_verified_citations`).
- `judge_answer` and `label_relevance` duplicated the entire `client.with_options(...).messages.parse(...)` + truncation-error-translation logic. Extracted to `call_structured_judge` (`src/eval/llm_call.py`).
- `daily_cost.py`'s sqlite functions expect a `Path`, but `report_usage` initially passed `ObservabilityConfig.cost_db_path` (a `str`) straight through, crashing on `.parent`. Caught within the same TDD cycle before it shipped.
- `generate_answer`/`verify_citations` initially called `report_usage` unconditionally, even under the no-observability default — writing real cost data to the real `data/daily_cost.sqlite3` on every test run, since `noop_observability()`'s default `ObservabilityConfig` still points at a real path with a real price table. Cost recording is now gated on whether `observability` was explicitly given, not on its defaulted value; same guard applied to `call_structured_judge`.
- `nightly-eval.yml`'s first two real runs failed with `httpx`/`h11` `LocalProtocolError: Illegal header value`, surfacing as `anthropic.APIConnectionError: Connection error`. Root cause: `.env`'s `ANTHROPIC_API_KEY=` line has had a stray leading space since Block 4 -- silently tolerated by `pydantic-settings`' dotenv parser locally, but preserved byte-for-byte by a raw shell pipe into `gh secret set`, landing a real leading space in the GitHub secret and thus the HTTP header. Fixed by piping through `xargs` (trims whitespace) when setting the secret; `.env` itself left untouched.
- **Block 7 audit (2026-07-16)**: `eval/pipeline.py`'s `_evaluate_one` — the eval harness's only real caller of `generate_answer`, `verify_citations`, and `judge_answer` — never passed the new `observability`/`observability_config` parameter to any of them, so every real Anthropic API call an eval run makes silently never counted toward `daily_cost_cap_usd`, despite each function's own cost-tracking being fully tested in isolation. `run_eval` now builds one `ObservabilityContext` and threads it through retrieval, rerank, generation, verification, and judging. Also extracted a duplicated `SpyTracer`/`_SpySpanCtx` test fake (hand-rolled independently across 4 test files, 6 tests) into a shared `spy_tracer` fixture (`tests/conftest.py`), and strengthened two existing span tests that captured `span.update()` calls but never asserted on their usage/cost contents.
- **Span nesting (HIGH, Block 9)**: every real query produced 6 separate root traces in Langfuse instead of one nested trace — sharing one `ObservabilityContext`/tracer instance (Block 7) was necessary but not sufficient, since nesting requires an active parent span and neither `answer_with_verified_citations` nor `answer_question` ever opened one. Invisible to Block 7's entire test suite (fakes asserted span names/counts, never real backend nesting); found on the first real trace screenshot, one block later. Fixed by wrapping each orchestrator's whole call chain in one parent span (`query.answer_with_verified_citations`, and the identical bug in `eval/pipeline.py`'s `_evaluate_one` via `eval.question.<id>`). Regression coverage added structurally: `spy_tracer` (`tests/conftest.py`) now records enter/exit order, catching a missing wrapping `with` block without needing a real backend.
- A same-day attempt to fix generation non-determinism by adding `generation.temperature=0.0` was itself a regression, shipped verified only against fake-client tests: `claude-sonnet-5` rejects `temperature` with a real 400 error ("deprecated for this model"), and a follow-up probe found `top_p`/`top_k` are deprecated for this model too — it exposes no manual sampling control at all. Reverted; a regression test now asserts `generate_answer` never sends any of the three. The eval harness's tolerance-band comparison (already in place since Block 6) remains the real mitigation for this non-determinism, not a stopgap.

### Known issues (tracked in `BUGS.md`)
- Sliding-window sizing still uses word count as a token-count proxy, which undercounts badly on numeral-dense text; 143 of 617 chunks exceed 650 tokens.
- `SIZE_RATIO_THRESHOLD` (subsection-header font-size heuristic) is still an untuned constant.
- Printed page labels (roman numeral / chapter-relative) are classified but not yet wired into chunk metadata.
- Table extraction is unhandled and unverified against the real corpus.
- `Settings` loads `config.yaml` relative to the cwd — running the CLI outside the repo root silently skips the corpus body-page-range filter (plus six smaller entropy items from the 2026-07-13 audit).
- Rerank latency on real chunks is ~5.3s per 20-candidate query on CPU (design claim of ~130ms holds only for short passages); chunks past the model's 512-token input limit are silently truncated during scoring. Block 4 decision item.
- Real-corpus spot-check found one design-doc sample question (short-field vs. soft-field takeoff) only half-answered — a retrieval/rerank surfacing gap, not a generation bug. Input for Block 6's golden dataset. (2026-07-14 spot-check: the same question answered both halves correctly on this run — non-deterministic retrieval/rerank, not confirmed fixed; no regression test pins either outcome.)
- `ObservabilityConfig.price_table` ships with placeholder pricing figures, not yet verified against Anthropic's real current published pricing.
- Real end-to-end Langfuse trace verification is blocked: `LANGFUSE_PUBLIC_KEY`/`LANGFUSE_SECRET_KEY` are still unset in `.env`. Every other Block 7 chunk is built and tested via fakes/`NoOpTracer`, but no real trace has ever been produced or inspected against the actual Langfuse Cloud dashboard.
- Citation verification (Block 5) only checks the citations `GeneratedAnswer` already listed, not full per-claim faithfulness of `answer_text` — an uncited hallucinated sentence isn't caught. `citations.judge_model`/`judge_temperature` duplicate `eval.judge_model`/`judge_temperature`'s default without sharing config. `citations/pipeline.py` re-fetches cited-chunk text rather than reusing Block 4's internal fetch.
- **HIGH**: Block 6's first real eval run caught generation fabricating an 8-item "According to the handbook" errors list (question 4) that isn't present in its single cited chunk or anywhere else in the corpus — a citation-scoped-generation failure Block 5's per-citation faithfulness check doesn't catch, since the cited chunk itself is genuinely relevant. Not yet fixed.
- `GenerationConfig` has no `temperature` field (defaults to the Anthropic SDK's default) — the same question (q1, VMC vs. VSO) scored `correct=True` on one run and `correct=False` on an immediate re-run, real answer-correctness non-determinism, not just retrieval/rerank variance.
- The eval response cache's `config_hash` only hashes `Settings` fields, not code logic — a metrics-code bug fix does not invalidate stale cached results with the same config.
- Design doc's "each trace and eval run is attributable to a specific prompt version" acceptance criterion is only half-met: Block 6's `EvalRunResult` carries prompt versions, but no Langfuse span carries `PROMPT_VERSION` in its metadata.
- `generate_answer`, `verify_citations`, and `call_structured_judge` each independently implement the same with-options/parse/catch-ValidationError/report-usage sequence (3 sites, not identical — two open a tracer span, one deliberately doesn't). Found at the Block 7 audit, deferred rather than fixed mid-audit pending a shared config shape decision.
- **Block 8 audit (2026-07-17)**: `cheap-gate.yml` had no explicit `permissions:` block (inherited the repo default) and no `concurrency` group despite triggering on both `push` and `pull_request` -- a PR branch's every commit would have fired two overlapping runs with nothing to cancel the stale one. Added `permissions: contents: read` and a cancel-in-progress concurrency group; `nightly-eval.yml` gets a non-cancelling group too, so overlapping triggers can't race on its baseline-commit push. Found by the audit's fresh-eyes review, re-verified passing on real GitHub Actions after the fix.
